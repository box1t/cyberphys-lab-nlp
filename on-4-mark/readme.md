
# 📄 Отчёт по исследованию эффективности техник prompt engineering

## 📌 Цель

Оценить влияние различных техник промптинга на качество классификации SMS-сообщений (spam / ham) с использованием LLM модели.

---

## 📦 Данные

Использован датасет:
**SMS Spam Collection Dataset**

* ~5500 сообщений
* Классы:

  * `0` — Ham (не спам)
  * `1` — Spam

Для эксперимента использована подвыборка сообщений.

---

## ⚙️ Методика

Для каждой техники промптинга:

1. Формировался системный prompt
2. Выполнялся inference через LLM
3. Получались предсказания (`verdict`)
4. Сравнивались с истинными метками
5. Рассчитывались метрики:

   * Accuracy
   * Precision
   * Recall
   * F1-score

---

## 🧠 Рассматриваемые техники

1. **Zero-shot** — без примеров
2. **Chain-of-Thought (CoT)** — с пошаговым рассуждением
3. **Few-shot** — с примерами
4. **CoT + Few-shot** — комбинация

---

## 📊 Результаты

| Technique      | Accuracy | Precision | Recall | F1-score |
| -------------- | -------- | --------- | ------ | -------- |
| Zero-shot      | 0.69     | 0.27      | 0.47   | 0.34     |
| CoT            | 0.62     | 0.23      | 0.53   | 0.32     |
| Few-shot       | 0.75     | 0.36      | 0.59   | 0.44     |
| CoT + Few-shot | 0.77     | 0.38      | 0.53   | 0.44     |

---

## 📈 Анализ результатов

### 🔹 1. Zero-shot

* Базовый уровень качества
* Модель работает без дополнительного контекста
* Низкий F1-score (0.34)

---

### 🔹 2. CoT

* Незначительное улучшение recall
* Но общее качество (F1) ниже zero-shot
* Причина: малая модель (0.5B) плохо использует reasoning

---

### 🔹 3. Few-shot

* Существенное улучшение всех метрик
* Особенно:

  * Accuracy: 0.75
  * F1: 0.44

👉 Модель адаптируется к задаче через примеры

---

### 🔹 4. CoT + Few-shot

* Лучшая accuracy: **0.77**
* F1 остаётся на уровне few-shot
* Reasoning даёт небольшой дополнительный эффект

---

## ⚠️ Важное наблюдение

Во всех техниках наблюдается:

```text
низкий precision (0.23–0.38)
```

👉 Это означает:

* модель склонна к **ложноположительным ошибкам**
* часто помечает обычные сообщения как спам

---

## 🧠 Интерпретация

Причины:

1. Малый размер модели (Qwen2.5:0.5B)
2. Ограниченные способности к обобщению
3. Смещение в сторону класса spam

---

## 🔬 Ограничения эксперимента

* Используется небольшая LLM модель
* Ограниченная подвыборка данных
* Нет fine-tuning

---

## 📌 Вывод

> Проведённое исследование показало, что техники prompt engineering существенно влияют на качество работы LLM в задаче классификации SMS.
>
> * Zero-shot демонстрирует базовый уровень
> * CoT не даёт значительного улучшения из-за ограничений модели
> * Few-shot значительно повышает качество
> * Наилучшие результаты достигаются при использовании CoT + Few-shot
>
> Таким образом, **наиболее эффективной техникой является few-shot подход**, а добавление reasoning даёт умеренный прирост.
>
> В целом, LLM могут быть применены для задачи фильтрации спама, однако требуют более мощных моделей или дополнительной настройки для достижения промышленного качества.

# 📄 Дополнение: Smoke Test (первичная проверка системы)

## 📌 Цель

Провести быструю проверку работоспособности LLM-сервиса и корректности интеграции компонентов:

* FastAPI
* Ollama
* LLM (Qwen2.5:0.5B)

---

## ⚙️ Методика

Для тестирования использован небольшой набор из **3 SMS-сообщений** с ручной разметкой:

* 1 сообщение класса *ham*
* 2 сообщения класса *spam*

Оценка производилась теми же метриками:

* Accuracy
* Precision
* Recall
* F1-score

---

## 📊 Результаты smoke test

```
Accuracy: 0.67

Class 0 (Ham):
Precision: 0.50
Recall: 1.00
F1-score: 0.67

Class 1 (Spam):
Precision: 1.00
Recall: 0.50
F1-score: 0.67
```

---

## 📈 Интерпретация

* Модель корректно определила все сообщения класса **ham** (recall = 1.00)
* Однако часть сообщений класса **spam** была пропущена (recall = 0.50)
* Общая точность составила **0.67**

---

## ⚠️ Ограничения smoke test

> Данный тест НЕ является репрезентативным.

Причины:

1. **Крайне малый объём данных (3 сообщения)**
2. Высокая чувствительность метрик к отдельным ошибкам
3. Отсутствие разнообразия примеров

---

## 🧠 Важное наблюдение

> Даже одно неверное предсказание существенно меняет метрики.

Например:

* 1 ошибка → accuracy падает до 0.67
* 2 ошибки → уже 0.33

---

## 🔬 Вывод по smoke test

> Smoke test показал, что система функционирует корректно:
> сервис обрабатывает запросы, модель возвращает предсказания, формат ответа соответствует требованиям.
>
> Однако полученные метрики не могут использоваться для оценки качества модели.
